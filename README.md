# OJ（面向程序设计的在线判题平台）

## 选题理由

这个选题具有较好的扩展性，基础版本可以完成在线做题、提交评测和比赛组织；进一步可以加入**高并发控制、防作弊系统、异常行为检测**、个性化练习推荐等高级功能。这个项目该题目既适合先实现基础版本作为课程设计或项目实践，也具备后续持续迭代和深入研究的空间。

## 项目建设目标

本项目拟设计并实现一个支持程序设计练习与竞赛管理的在线判题系统。系统面向**普通用户**、**参赛选手**和**管理员**三类角色，提供题目浏览、在线编程、代码提交、自动判题、比赛参与、实时榜单、防作弊监控和行为分析等功能。

项目最终希望达到的程度大致如下：

能够支持用户完成从注册登录、查看题目、编写代码、提交评测、查看结果的完整闭环。

能够支持管理员完成题目添加、比赛创建、用户管理、提交记录管理等后台功能。

能够通过Judge0实现基础自动判题，支持多测试点评测，并返回运行时间、内存占用、编译结果等信息。

能够在比赛过程中通过WebSocket实时推送判题结果和榜单更新，保证交互体验。

能够记录用户在答题过程中的异常页面切换等行为，形成基础防作弊日志。

在后续版本中，基于用户提交记录、答题时间、错误次数、页面切换次数等数据，对用户行为进行分析，构建初步画像，并尝试识别异常模式。

## 具体需求补充

### 1. 用户端需求

用户端不仅需要提供基础的做题功能，还要尽可能贴近真实OJ平台的使用体验。

用户登录后，可以查看题目列表、根据题目标题或难度筛选题目、进入题目详情页查看题目描述、输入输出说明、样例和限制条件、查看提交记录以及提交结果。进入做题界面后，用户可以直接在网页中编写代码，支持多种语言。

系统还包含比赛功能，那么用户能够进入比赛列表页，查看比赛题目，比赛榜单。

### 2. 管理端需求

管理端的核心目标是让系统具备可维护性和可运营性。

管理员能够管理用户信息和用户账号（包括封禁非法账户、修改密码等），还能管理题目（包括新增题目、编辑题目、删除题目、上传测试数据、设置时间与内存限制、指定支持语言）。

在比赛管理方面，管理员应能够创建比赛、设置标题、起止时间、题目列表、比赛类型和是否封榜。

管理员还应能够查看防作弊日志，例如用户在比赛期间频繁切换页面、长时间离开页面、短时间高频提交等异常行为。

### 3. 判题功能需求

判题模块支持以下能力：

支持多语言提交。

多测试点判定。

区分编译错误、运行时错误、答案错误等不同结果。

保证在**多人如统一考试场景**同时提交时系统仍然稳定。

### 4. 防作弊需求

第一阶段可实现的需求包括：监听页面失焦、标签页切换、页面隐藏与重新显示、频繁离开页面等行为，并将这些行为记录到日志表中。对于管理端，可以按比赛、用户、时间查看这些异常记录。系统后续可考虑扩展检测复制粘贴次数、异常提交频率、同时间段多账号行为相似性等内容。

### 5. 用户画像与异常行为分析需求

基于用户在平台上的长期行为，抽取出一定规律。例如：某用户更擅长哪类题目、平均做题时间如何、错误类型集中在哪些方面、是否存在极端异常行为等。

## 各模块的大致实现思路

### 1. 前端实现思路

前端采用Vue3进行开发，主要负责页面展示、用户交互。判题结果和比赛榜单则通过WebSocket推送，避免页面频繁轮询，提高实时性和用户体验。

前端在页面设计上分为用户端和管理端两套视图。

### 2. 后端实现思路

后端采用 Java Spring Boot 框架进行开发，主要负责系统业务逻辑处理、接口提供、权限控制、数据库交互以及与 Judge0 判题服务的通信。后端对外提供统一的 RESTful API，包括用户注册登录、题目查询、代码提交、比赛管理、提交记录查询、防作弊日志查询等接口，为前端页面和管理端功能提供完整的数据支持。

未来若系统面临更高性能挑战，可以考虑从两个层面优化判题核心。首先，评估并集成性能更优的C++沙箱组件（如SafeExec）以替换默认的Isolate，实现更精细的资源限制。其次，若现有架构无法满足高并发需求，将使用C++重构判题调度、资源监控等核心逻辑，或开发独立的C++微服务，以降低延迟并提升吞吐量。”

### 4. 判题系统实现思路

系统采用Judge0作为底层判题引擎。Judge0本身支持代码编译与运行，并能返回标准输出、错误输出、运行时间、内存使用和状态码等信息。后端通过HTTP接口调用Judge0，将用户提交的代码、测试数据、语言类型和资源限制发送出去，再接收其返回结果。

为了支持**多测试点**判题，可以在系统中维护测试数据文件，并由后端按题目配置组织评测任务。对于需要批量导入的题目，可通过FPS格式XML文件解析出题目信息、样例数据和测试点配置，并自动生成数据库记录和测试数据文件目录。

为了避免**提交量较大**时阻塞主线程，系统引入**Redis队列**。代码提交后，并不一定要立刻同步等待判题完成，而是可以先将提交任务写入Redis，由独立的消费进程逐步调用Judge0处理，处理完成后再将结果回写数据库并推送给前端。这样可以提高系统在多人同时提交时的吞吐能力和稳定性。

### 5. 用户画像实现思路

用户画像功能主要建立在数据库中已有的提交记录和行为日志之上。系统可以定期统计用户的历史数据，例如总提交次数、AC次数、平均单题尝试次数、平均作答时长、不同题型通过率等，并将统计结果展示在管理端或用户个人页面中。

对于异常行为分析，可以先采用规则型方法实现，例如：

当比赛期间标签页切换次数过多、提交间隔异常短、短时间内通过率突然大幅上升时，系统标记该用户为可疑用户。

## 具体开发计划

### 1.0 基础功能版本

完成系统前后端基础框架搭建，实现用户注册登录、题目管理、题目展示、在线编程、代码提交、判题结果返回等核心流程；完成用户端与管理端主要页面的开发；设计并建立用户表、题目表、提交记录表、比赛表等核心数据库表；实现后端对Judge0的基本调用，完成单次提交到结果展示的完整业务闭环。

### 2.0 优化与比赛版本

在基础判题流程可用的前提下，引入Redis队列优化提交处理逻辑，提升多人同时提交时的系统稳定性；增加比赛管理模块，实现比赛创建、比赛题目关联、比赛时间控制、实时榜单展示和封榜机制；前端通过WebSocket实现判题结果与榜单的实时推送；完成防作弊基础功能，能够识别并记录用户在比赛过程中切换标签页、离开页面等异常行为。

### 3.0 数据分析版本

在已有提交记录和防作弊日志的基础上，对用户答题行为进行统计分析，构建初步用户画像，包括做题数量、通过率、答题耗时、常见错误类型等信息；探索异常行为识别规则，对比赛期间可疑操作进行标记和预警；为后续个性化训练推荐和智能监管提供数据基础。
